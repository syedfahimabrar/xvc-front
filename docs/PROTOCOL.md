# Wire Protocol — X-VC Streaming Server

Verified against the working Hear-Me-Out X-VC service (`docs/reference/hearmeout-xvc-server.py`)
and its browser client. The MeanVC engine shares the same contract (differences noted at the end),
so a client written to this spec works with both engines unchanged.

All endpoints live on one port (default **5002**, env `MEANVC_PORT`), HTTPS/WSS.

---

## 1. Register a target voice

```
POST https://<host>:5002/api/meanvc/load-target
Content-Type: multipart/form-data
```

- Exactly one file field named **`wav`** containing a WAV file of the target speaker
  (a few seconds of clean speech; server processes at 16 kHz mono internally).
- Success `200`:

  ```json
  { "target_id": "3f2a…(32 hex chars)", "duration_seconds": 7.43 }
  ```

- Errors: `400` `{"error": "missing 'wav' file"}`; `500` `{"error": "<message>"}`.
- `target_id` is an in-memory handle (`uuid4().hex`). **It does not survive a server
  restart** — clients must be prepared to re-upload and retry on "Unknown target_id".
- Max upload size on the reference server: 10 MB.

## 2. Stream conversion

```
GET wss://<host>:5002/api/meanvc/stream?target_id=<id>&source_sr=<int>[&steps=<int>]
```

Query params:

| param | meaning |
|---|---|
| `target_id` | handle from `load-target` (required) |
| `source_sr` | sample rate of the PCM the client will send; server resamples to 16000 if different |
| `steps` | MeanVC only (flow-matching solver steps, default 2). **X-VC ignores it** — always safe to send |

### Handshake

The **first frame from the server is JSON** (text):

- ready: `{"status": "ready"}` — MeanVC additionally includes `"chunk_size": <int>`
- failure: `{"error": "Unknown target_id: <id>"}` followed by close

Clients MUST wait for `status == "ready"` before sending audio.

### Client → server

Binary WebSocket frames of **raw little-endian float32 mono PCM**, values in [-1, 1],
at `source_sr`. **No header, no tag bytes, any chunk length** — the server buffers
internally and runs a conversion window whenever enough look-ahead audio exists.
(The reference browser client sent 2048-sample chunks at 16 kHz ≈ 128 ms; smaller,
more frequent chunks reduce latency slightly.)

### Server → client

Binary WebSocket frames of **converted raw little-endian float32 mono PCM at 16 kHz**,
no tag bytes. Output arrives in bursts of the server's "current" window size
(default **120 ms = 1920 samples**, one or more per input feed). The client needs a
small jitter buffer before playback (see `docs/MAC_APP.md`).

### Session behavior

- One WebSocket = one conversion session (fresh window state, cross-fade tail, buffers).
- There is no mid-stream "reset" on X-VC — reconnect to reset. (MeanVC accepts a JSON
  text frame `{"action": "reset"}`.)
- Output audio lags input by the algorithmic look-ahead (~240 ms) **plus** GPU compute
  and network time — see `docs/PERFORMANCE.md` for the full budget.
- The server sends nothing while the client is silent only if the client stops sending;
  if the client streams silence, the server converts silence (send continuously —
  gating/VAD is a client-side choice with trade-offs for the cross-fade continuity).

## 3. TLS & auth (current state → what to build)

- **Current KTH server: self-signed cert** (`CN=*`, 10-year,
  generated by openssl at deploy). Dev clients must disable certificate validation for
  this host (e.g. `URLSessionDelegate` trust override in Swift). Do not ship that
  bypass outside dev builds.
- **No authentication exists** on the current server.
- The new standalone backend must add:
  - a static **bearer token**: `Authorization: Bearer <token>` on `load-target`, and
    `?token=<token>` on the WebSocket URL (browsers/URLSession can't set WS headers
    reliably); server reads it from env `XVC_AUTH_TOKEN`, rejects with 401/4401 close.
    **Implemented** in `server/xvc_server.py`: constant-time compare, checked before the
    WS handshake so a bad token gets a plain HTTP 401; unset `XVC_AUTH_TOKEN` = open server
    (logged loudly). Clients send it via `XVC_TOKEN` / `--token`.
  - real TLS if the box gets a hostname (Let's Encrypt), else documented self-signed
    with cert pinning in the app.

## 4. Endpoint NOT to use

The reference server also has `GET /api/meanvc/chat-proxy` — a Hear-Me-Out-specific
bridge that forwards converted audio to a dialogue model (PersonaPlex) using tagged
frames (`0x01` Opus, `0x03` converted PCM). **Irrelevant to this project; drop it in
the trimmed backend.** Only `load-target` + `stream` are needed.

## 5. MeanVC compatibility summary

| Aspect | X-VC | MeanVC |
|---|---|---|
| `load-target` | identical | identical |
| `stream` query params | `target_id`, `source_sr` | + `steps` (default 2) |
| ready frame | `{"status":"ready"}` | + `"chunk_size"` |
| PCM framing both directions | identical (float32 mono, 16 kHz out) | identical |
| text control frames | none | `{"action":"reset"}` |

Client rule of thumb: always send `steps`, tolerate extra keys in the ready frame,
treat any `{"error": …}` text frame as fatal for the session.
